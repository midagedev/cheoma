# 길가 배수로와 대문 건넘

> - **상태**: 현재 계약 + 리서치
> - **조사일**: 2026-07-24
> - **범위**: 조선 한양의 법정 도로 도랑, 발굴된 도심 배수로, 전통마을의 제한적 물길을 근거로 한
>   정적 길가 배수 계획. 실시간 강우 수리·침수 시뮬레이션은 포함하지 않는다.

## 1. 결론

조선의 길 옆 도랑은 근거가 있지만 **모든 시대·모든 마을길의 보편 장식**으로 만들 수는 없다.

- 『경국대전』을 인용한 국사편찬위원회 해설은 **한양 성내** 대로·중로·소로 양쪽에 너비 2척,
  환산 약 62cm의 도랑·배수구를 두도록 했다고 설명한다. 도랑 훼손·오물 투기를 처벌하고 파손된
  길과 다리를 수선했다는 관리 규정도 전한다.
- 서울역사박물관 인사도시유적전시관은 16세기 문화층에서 실제 배수로를 보존한다. 이 배수로는
  낮은 종로·시전행랑 주변 침수를 막았고, 시전·피맛길·점포와 배후 주거지 사이의 공간 경계로
  이어졌다.
- 국가유산청은 외암마을의 계곡물을 끌어들인 인공 수로를 생활용수·방화수·정원수에 쓴
  **다른 마을에서 보기 어려운 독특한 요소**라고 명시한다. 이것은 전통마을 전체에 상시 흐르는
  수로를 복제할 근거가 아니라 별도 지역 변형이 필요하다는 반증이다.

따라서 제품 기본값은 `배수로 없음`이다. Hanyang 계획도로와 capital의 제한된 간선도로 중 실제로
더 낮은 유출구까지 이어지는 구간에만 얕은 측구를 두고, 일반 hamlet/village/town에는 두지 않는다.
외암형 생활·방화 수로는 향후 독립된 지역 변형이며 길가 빗물 측구와 같은 타입으로 합치지 않는다.

## 2. 원문 사실과 적용 한계

### 2.1 한양의 법정 양측 도랑

국사편찬위원회 우리역사넷 「조선 초기의 도로」는 『경국대전』의 한양 도로 폭과 함께 길 양쪽
도랑을 2척으로 적고 약 62cm로 환산한다. 도로 침식, 도랑 훼손·오물 투기, 길·다리 수선도 같은
도시 관리 문맥에서 설명한다.

이 수치는 **한양 성내의 법정 규범**이다. 현존 촌락의 평균 실측치도, 모든 구간이 정확히 그
폭으로 시공·존속했다는 조사도 아니다. 생성기에서는 Hanyang/capital 계획가로의 유형 근거와
상한 감각으로만 쓰며 지방 고샅에 0.62m를 일괄 복제하지 않는다.

### 2.2 16세기 인사동 배수로 유구

서울역사박물관은 인사동 유적의 16세기 문화층에서 공동우물·배수로·주거지를 확인하고,
배수로가 지대가 낮은 종로와 시전행랑 주변의 침수를 예방했다고 설명한다. 배수로를 기준으로
상업지역과 배후 주거지가 나뉘고, 좁은 건물들이 그 선을 따라 배치된 사실도 확인된다.

이는 도심 배수로가 단순한 표면 장식이 아니라 **낮은 지형과 토지 조직을 잇는 연속 시설**이었음을
보여 준다. 그러나 전시 해설은 전국 도랑의 단면, 재료, 경사, 대문 앞 덮개 형식을 제공하지 않는다.
제품은 유출구 없는 짧은 홈을 무작위로 뿌리지 않되, 이 유구 하나에서 전국 표준 형상을 추론하지 않는다.

### 2.3 외암마을 인공 수로는 별도 유형

국가유산청의 외암마을 해설은 상부 계곡물을 살림집까지 끌어 생활용수·방화수·곡수·연못에
사용한 인공 수로를 마을의 독특한 특성으로 든다. 물이 보이는 생활 수로는 빗물만 받는 측구보다
기능과 수원이 넓고, 외암의 풍수·정원·방재 문맥에 결합돼 있다.

따라서 외암을 근거로 모든 마을 배수로에 항상 물을 흐르게 하지 않는다. 이번 범위에는 취수원,
저수, 방화수, 정원 급수, 생활용수 사용을 넣지 않는다.

### 2.4 대문 건넘의 증거 공백

확인한 공식 자료는 도랑과 도로·주거의 관계를 보여 주지만, 일반 촌락의 대문마다 어떤 재료와
치수로 작은 건넘을 만들었는지 전국적으로 정리하지 않는다. 생성기의 대문 앞 짧은 건넘은
**보행 접근 계약을 보존하기 위한 제품 해석**이다.

가장 절제된 기본형은 현장에 이미 있는 자연석 어휘를 쓰는 낮은 판석 2~3매다. 난간·홍예·기념적
교각을 만들지 않고, 도랑을 대문 전체에서 없애지도 않는다. 판석 수, 폭, 두께는 역사 평균이 아니라
대문 통과 폭과 충돌 안전에서 정한다. 향후 특정 마을의 실측·수리보고서가 확인되기 전에는
목교·석판교·암거를 신분 또는 지역별 고증 타입으로 이름 붙이지 않는다.

## 3. 제품 계획 계약

### 3.1 하나의 순수 계획

렌더러보다 먼저 Three·DOM 없는 계획이 다음을 결정해야 한다.

- stable road ID와 segment ID
- `none | roadside-ditch` 유형과 길의 왼쪽/오른쪽 면
- 시작·끝, 내리막 방향, 유효 유출구
- 노면 가장자리에서의 오프셋, 상부 폭, 깊이
- 대문 접근과 만나는 `gate-crossing` 구간
- 도로 junction·개울·대하천·논·필지·성문 주변의 끝처리

같은 seed·도로·지형·필지는 sync/Worker/fallback에서 같은 JSON record를 내야 한다. 렌더러가
카메라 거리나 프레임마다 도랑 위치를 다시 판단하지 않는다.

### 3.2 선택 규칙

1. **Hanyang 계획가로**: 법정 한양 문맥을 적용할 수 있는 대로·중로·소로는 최대 양측 측구를
   허용한다. 각 면이 실제 지형의 더 낮은 출구까지 이어지지 않으면 그 면은 생략한다.
2. **Capital 간선도로**: 대로·중로 중 낮은 지형, 시전 배후 경계, 성문 접근처럼 모이는 물의
   목적지와 내리막 출구가 분명한 구간의 한쪽만 선택한다. 인사동 유구를 아무 시장의 동일 평면으로
   복제하지 않는다.
3. **일반 hamlet/village/town**: 이번 제품 계약에서는 `none`이다. 공식 자료가 뒷받침하지 않는
   촌락 빈도나 지역별 단면을 절차 확률로 추측하지 않는다.
4. **외암형 수로**: 이번 `roadside-ditch` 선택 확률에 넣지 않는다. 수원과 생활·방화·정원 기능을
   함께 계획할 수 있을 때 별도 archetype으로 다룬다.

도로 길이만으로 도랑을 뽑지 않는다. 내리막 방향과 유출구를 찾을 수 없으면 `none`으로
fail-closed한다.

### 3.3 단면과 경사

한양 2척은 도시 규범의 근거이고 농촌 수치가 아니다. 초기 제품 단면은 다음처럼 **명시적인
가독성·충돌 안전 범위**로 둔다.

- 상부 폭은 도로 폭의 작은 비율로 계산하되, 한양의 약 0.62m를 모든 선택 구간의 고정값으로 쓰지 않는다.
- 깊이는 사람이 빠지는 절벽이나 성곽 해자가 아니라 맨흙·잔돌로 마감된 낮은 V/U형 측구로 제한한다.
- 중심선 높이는 실제 삼각분할 지형을 읽고 하류 방향으로 역구배가 생기지 않게 단조화한다. v1은
  공용 지형 격자를 도랑마다 다시 세분하거나 boolean 절삭하지 않는다. 대신 보이는 바닥을 실제 지형보다
  12mm 위에 두고, 최대 2.5cm의 상류 보정 안에서만 단조 경사를 만들며, 그 위에 4cm 이하의
  낮은 U자 입술을 세운다. 단면 바깥 rail은 같은 얕은 bed plane으로 되붙이고 1m마다 실제
  삼각 지형을 다시 읽어 긴 chord가 지표 아래로 파고들지 않게 한다.
  이 범위로 성립하지 않는 구간은 도랑을 띄우거나 지형 아래 숨기지 않고 생략한다.
- 장거리 직선 홈 대신 도로 polyline의 완만한 곡률을 따르며 junction에서 서로 겹치거나 막힌
  쐐기를 만들지 않는다.

정확한 폭·깊이 clamp는 역사 실측값으로 표기하지 않고 브라우저 근경의 보행 가독성, triangle 예산,
도로 corridor 충돌을 함께 보고 정한다.

### 3.4 대문 건넘

측구가 `parcel.access.gateEdge/gateT`에서 도로 접근선을 가로지를 때만 건넘을 만든다.

- 건넘 중심은 추정한 남향 정면이 아니라 저장된 실제 도로측 대문 접근점이다.
- 유효 통과 폭과 작은 여유만 덮고, 인접 도랑은 계속 보이게 한다.
- 낮은 자연석 판석 2~3매를 단일 `stone-slab` record로 표현한다.
- 판석 상면은 도로/대문 landing과 급단차가 없고, 아래 도랑 바닥을 관통하지 않는다.
- 집마다 장식 교량을 만들거나 난간·기와지붕·석수·홍예를 붙이지 않는다.

건넘은 도랑의 존재에 종속된다. 도랑이 생략되면 별도 판석 장식도 만들지 않는다.

## 4. 렌더·성능·모드 경계

- 도랑은 월드 공간의 낮은 U자 물리 geometry·법선·깊이를 사용한다. 바닥 전 폭이 공용 terrain보다
  명시적인 12mm 위에 있어 첫 표면이 되고, 양쪽 4cm 입술과 얕은 bed plane으로 되붙는 바깥 사면
  사이가 얕은 홈으로 읽힌다. 노면 위 검은 선
  texture나 카메라 의존 `polygonOffset`은 쓰지 않는다. 실제 terrain boolean 절삭을 생략하는 이유는
  0.48m 측구 때문에 수백 m 공용 지형 격자를 국소 재분할·직렬화하지 않고도 동일한 색상·DoF·수묵
  depth를 한 물리 단면으로 보존하기 위해서다.
- 같은 단일 재질과 vertex color를 사용하는 모든 segment와 판석을 각각 정적 병합한다.
  도랑·집 수에 비례한 material/program/texture를 만들지 않는다.
- 기본 측구에는 물 mesh, 흐름 animation, 반사 pass, 입자, foam을 두지 않는다. 비가 내려도
  수위·유량·침수·토사 이동을 실시간 계산하지 않는다.
- 계절·날씨는 기존 지형 팔레트와 조명 결과만 공유한다. `rain` 상태가 계획 record나 충돌 경계를
  바꾸지 않는다.
- focus와 aerial은 같은 record를 본다. 원경에서는 별도 impostor를 만들지 않고 작은 내부 면이
  자연스럽게 사라지며, 물리적 지형 실루엣과 대문 접근만 유지한다.
- 수묵·DoF·shadow depth는 일반 불투명 지형 규칙을 따른다. 별도 투명 depth owner를 추가하지 않는다.

## 5. 검증 기준

### 순수 계약

- 같은 입력의 byte-stable record와 전역 `Math.random` 비사용
- 유출구 없는 길, 평탄/역경사 구간, 너무 짧은 구간의 `none`
- Hanyang 최대 양측, capital 간선 한쪽, 일반 hamlet/village/town 없음 정책의 분리
- 모든 ditch 중심선의 하류 단조성, 도로 corridor 내부 bounded footprint
- junction·stream·river·paddy·city-gate 끝처리와 자기교차 0
- 대문 접근을 막지 않는 건넘 1개, 도랑 없는 대문의 건넘 0개

### 제품·브라우저

- 같은 seed·카메라의 도랑 ON/OFF 근경과 aerial PNG 직접 비교
- 대문을 클릭·focus할 때 판석이 문길을 막거나 카메라를 가리지 않는지 확인
- 맑음/비/눈 전환에서 geometry·material·program·draw identity 불변
- 도랑 없는 일반 마을과 도랑 있는 Hanyang/capital 간선도로를 함께 확인
- sync/Worker/fallback scene hash와 production References UI의 출처·활용·한계·링크 확인

### 구현 결과

- `src/village/drainage-plan.js`가 도로·필지·논·실제 삼각 지형에서 `surfaceY`까지 담은 불변
  world-space `runs`/`crossings`를 만들고 `src/api/drainage-plan.js`가 Three 없는 재사용 경계를 제공한다.
- `src/village/drainage-geometry.js`는 모든 도랑을 한 indexed mesh, 모든 판석을 한 merged mesh로
  조립한다. 한 계획은 최대 두 mesh, 한 texture-free material family, shadow caster 0을 유지한다.
- `src/village/plan.js`가 후속 RNG 소비가 끝난 뒤 이 계획을 붙이고 `populate.js`는 계획을 다시
  해석하지 않는다. 빈 농촌 계획은 renderer와 material을 만들지 않는다.
- `npm run check:drainage`가 정책·경사·지형·도로·수계·생산지·대문축·불변/직렬화 계약을,
  `npm run shoot:drainage`가 실제 capital 제품의 동일 카메라 ON/OFF, 자원 예산, 반복 plan/geometry
  hash와 근경·부감 PNG를 검사한다. 이 브라우저 증거는 `check:surface:browser`에 포함된다.
