# 일반 주택 다양성 계약

**상태: 계약.** 이 문서는 일반 기와집·초가의 필지, 살림 위계, 평면, 실제 처마 적합성을 하나의 결정론적 정책으로 유지하기 위한 기준이다. 특정 문화재나 한 지역의 민가 분포를 복원하는 문서가 아니다.

## 자료를 구현 규칙으로 옮기는 범위

국가한옥센터(AURI)의 [「한옥의 종류」](https://www.hanokdb.kr/theology/sub_02)는 ㅡ자를 기본적이고 널리 쓰인 평면, ㄱ자를 ㅡ자를 직각으로 꺾은 평면, ㄷ자를 ㅡ자 몸채에 두 날개가 붙는 평면, ㅁ자를 작은 안마당을 에워싸는 평면으로 설명한다. 전남·경남의 ㅡ자, 중부의 ㄱ자, 영남 북부 상류주택의 ㄷ자, 안동·한랭지의 ㅁ자 같은 지역 사례도 함께 든다. 그러나 현재 생성기에는 검증된 지역 축이 없으므로 이를 전국의 엄격한 분포표로 일반화하지 않는다. 일반 주택은 **ㅡ·ㄱ·ㄷ을 가중 레퍼토리**로 쓰고, ㅁ자는 이 무작위 레퍼토리에서 기본 off를 유지한다. 안동문화권·겨울 바람·가계 맥락을 호출자가 명시한 경우에만 종가 한 채의 별도 순수 계획을 시도하며, 메타데이터를 확률이나 신분 판정에 쓰지 않는다. 세부 근거와 fit·접근·일조 경계는 [`mja-hanok.md`](mja-hanok.md)를 따른다.

국사편찬위원회 우리역사넷의 [가사·가대 제한 해설](https://contents.history.go.kr/front/km/view.do?levelId=km_033_0050_0020_0010)은 1395년 집터 지급·제한과 1431·1440년 신분별 가옥 규모 제한을 설명한다. 문헌의 `35부 ↔ 2부`, `60/50/40/30/10칸`은 **법적 지급량·상한과 규범**이지 실제 주택의 평균, 빈도, 선형 비례를 뜻하지 않는다. 생성기는 숫자를 17배 필지나 신분별 고정 칸수로 복제하지 않고 다음의 단조 방향만 사용한다.

- 필지와 살림 여력이 클수록 큰 집·높은 처마·복합 평면의 가능성이 커진다.
- 낮은 위계의 작은 필지는 단순하고 낮은 평면이 우세하다.
- 이 관계는 확률 가중치이지 신분을 외형 하나로 단정하는 분류표가 아니다.

## 단일 정책과 형태 예산

`src/village/house-diversity.js`는 Three와 전역 난수에 의존하지 않는 정책이다. 입력은 실제 `plotW × plotD`, `rank`, 같은 집의 상관 변수 `wealth`, 마을 규모다. 도로변·내부 필지는 이미 가진 density `structureScale`에서 규모를 파생하고, 그 값이 없는 위성 부락만 `settlementScale01`을 저장해 중복 필드를 피한다. `assignFittedVariationSequence()`가 최초 배치와 focus 단건 재굴림에서 모두 이 정책을 호출하고, 선택된 `planShape/bays`는 `variantOv()`를 통해 FULL 빌더, MID, FAR impostor, 편집 명세가 함께 읽는다.

같은 상관 점수는 형태마다 따로 해석하지 않고 먼저 하나의 불변 `householdCompositionPolicy()`로 줄인다. `.28/.50` 경계는 사료에서 읽은 가구 빈도가 아니라, 작은 필지에서 큰 평면을 억제하면서 정착지 위계가 올라갈수록 복합형이 자연스럽게 늘도록 정한 제품 보정값이다. 아래 수치는 상한이나 강제 매핑이 아니라 seed-local 인접 변주가 모이는 **중심값**이다.

| 살림 단계 | 대표 본채 칸수 | 대표 연결 날개 |
| --- | ---: | ---: |
| small | 3칸 | 0 |
| medium | 4칸 | 1 |
| large | 5칸 | 2 |

칸폭·문 높이를 통째로 확대하지 않고 **칸수와 연결 채 수**로 큰 살림을 표현한다. 기와집은 ㅡ·ㄱ·ㄷ로 날개 `0/1/2`를 모두 실현하고, 한 몸채인 초가는 같은 단계의 `3/4/5칸`만 소비한다. 이웃집은 같은 단계에서도 seed-local 확률로 인접 평면을 고를 수 있고, 실제 처마 fit은 언제나 큰 후보를 더 작은 후보로 내릴 수 있다. 독립 별채는 이 본채 위계의 근거나 사료상 빈도로 세지 않고 기존 요청 확률도 늘리지 않는다. 다만 승인된 한 채는 `auxiliary-building-plan.js`가 생성된 실제 처마 또는 commit된 편집 처마·필지·대문 접근·30° 일조·마당 hard object를 통과한 좌표와 원래 크기로 한 번 계획하며, 같은 물리 root가 FULL/MID/FAR 전 구간에 남는다. 이 별채의 실제 방향·높이 volume은 근경 카메라 후보에도 장애물로 들어간다.

필지 생성기는 이미 도로·지형·위계에 따라 연속 치수를 만들며, 고정 town fixture의 일반 주택은 69필지 중 폭 66종·깊이 66종, 면적 `70.3–281.2㎡`(약 4배)를 가진다. 이 정책은 그 필지를 다시 흔들어 도로·담·일조 계약을 깨지 않고, 실제 크기 차이가 평면·높이·폭·색으로 읽히게 만든다.

기와집의 활성 prototype group은 성능 예산상 **항상 4개**다.

| 그룹 | 평면 | 칸 계약 | 용도 |
| --- | --- | --- | --- |
| 1 | ㅡ | 3칸 | 작은 필지·낮은 살림에서 우세 |
| 2–3 | ㄱ 좌·우 | 4칸 | 중간 범위의 기본 반가 어휘 |
| 4 | ㄷ | 제품 prototype 5칸, 범용 하한 4칸 | 넓은 필지·상위 살림의 드문 복합형 |

색 차이는 새 재질을 만들지 않고 기존 역할별 `instanceColor`에 싣는다. 새 평면 때문에 material variant나 활성 InstancedMesh 그룹을 늘리지 않는다.

서로 다른 평면은 분해된 부재 수와 순서가 다르므로 배열 인덱스로 재질을 공유하지 않는다. `materialShareKey()`가 팔레트 의미, 색·광택, 텍스처 반복·offset·wrap을 모두 비교해 같은 표면만 공유한다. 따라서 ㅡ·ㄷ도 기준 ㄱ 팔레트를 재사용하지만, 치수에서 파생된 기와 반복과 변주별 창호 무늬는 자기 재질을 보존한다. 고정 브라우저 fixture는 형태별 부재 수가 달라도 의미가 같은 표면을 기준 팔레트와 공유하고 전체 상한을 재질 120개·텍스처 60개로 둔다.

편집 패널도 선택된 형태를 따른다. ㅡ자에는 효과 없는 날개 길이·폭을 숨기고, 본채 폭은 칸수×칸폭에서 계산한 범용 실제 하한(기본 3칸 `3.3m`, 직접 편집한 ㄷ 최소 4칸 `4.4m`)부터 시작한다. 자동 배치 ㄷ prototype은 위계가 읽히도록 5칸을 쓴다. ㄱ 좌우 소유권은 `variantMirrorX()` 한 곳에서 FULL/MID/FAR와 focus/edit 오버레이에 함께 전달한다. 기와↔초가 유형 전환은 이전 유형의 숨은 값을 넘기지 않고 대상 기본 spec으로 다시 시작하며, 코어가 clamp한 최종 spec을 패널 값에도 되돌린다.

## 실제 처마가 최종 권위다

ㄷ자는 명목상 큰 집이라는 이유만으로 채택되지 않는다.

1. 필지가 최소 `175㎡`, 폭 `13m`, 깊이 `12m`이고 composition이 `large`여야 후보가 된다.
2. `src/layout/giwa-footprint.js`가 ㄷ자의 칸수를 최소 4로 정규화한다. 3칸 ㄷ자는 어떤 호출 경로에서도 성립하지 않는다.
3. `impostorHouseSpec()`가 FULL과 같은 본채+두 날개 지붕 직사각형을 만든다.
4. `fitHouseWithinParcel()`이 모든 실제 처마 모서리를 부정형 필지 안쪽 `0.3m`에 넣는다.
5. ㄷ자는 fit factor `0.82`, 최종 유효 scale `0.76`보다 작아지면 장난감처럼 축소하지 않고 `ㄱ → ㅡ`로 내린다. ㄱ도 맞지 않으면 ㅡ로 내리고, 그래도 실패하면 필지 후보를 버린다.

렌더러에서 위치·회전·축척을 보정해 이 계약을 우회하지 않는다. 필지·집·담·픽킹은 `frontDir` 하나를 유지하며, 주택 간 비겹침은 서로 겹치지 않는 필지와 그 안의 실제 처마 여유에서 파생한다.

## 검증

- `npm run check:house-diversity`: 불변 composition 중심값, 기존 prototype 4개 안의 3·4·5칸/0·1·2날개 위계, 필지 면적 3.5배 이상·연속 치수 다양성, ㅡ·ㄱ·ㄷ 레퍼토리, 작은 필지 ㄷ 금지, ㄷ 최소 4칸, 기와·초가 paired seed cohort에서 실제 처마 폭·깊이·높이·칸수·연결 날개의 단조 효과, 형태별 유효 편집축, 실제 처마 0.3m 여유와 이웃 처마 비겹침.
- `npm run check:layout`: 전 규모·다중 seed에서 실제 variant 처마, 일조, 도로·개울·논·식생 이격.
- `npm run check:lod`: FULL/MID/FAR가 같은 ㅡ·ㄱ·ㄷ 몸체·지붕 수를 쓰고 ㄱ 좌우 반전을 보존하는지 검사.
- `npm run check:auxiliary-building`: 별채 요청을 축소하지 않고 안전한 경우만 승인하며, 불변 footprint와 빌린 재질의 48-triangle geometry가 일치하는지 검사.
- `npm run check:worker`: sync·실 Worker·fallback 결과가 같은 계획과 picking 계약을 유지하는지 검사.
- `npm run check:app`: 실제 focus/edit에서 ㄱ 좌우와 기와↔초가 target-default/clamp 동기화가 유지되고 ㅡ/ㄷ 재질·텍스처 공유 상한을 지키는지, 참고 자료 모달이 이 규칙의 기관·제목·활용·URL·라이선스를 실제 제품 UI에 렌더하는지 검사.
